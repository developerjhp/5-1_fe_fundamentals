## PR 본문 초안

### 1. 필터링 전략

서버 사이드 렌더링을 선택했습니다. 초기에 1회 받아오고 런타임에 필터링하는 클라이언트 사이드 렌더링의 경우, 특정 상품이 삭제/비활성화/수정 되더라도 즉각 반영되지 않기 때문에 고려하지 않았습니다.

또한, URL 쿼리 파라미터로 관리하는 키 값들이 API 파라미터와 동일해 흐름상, 구현상 복잡도가 적다고 판단했습니다.

### 2. 상태 관리 전략

`search`, `categories`, `sort`를 URL 쿼리스트링으로 관리하고, 상품 목록 조회도 그 값을 그대로 사용합니다. 각 컴포넌트에서 URL을 읽기 때문에 URL 자체가 SSOT로 적용되고, 어떤 컴포넌트에서 참조하든 동일한 값을 유지할 수 있습니다.

이를 위해 `useRouteParams` 훅을 만들어서 URL을 읽고(`currentQuery`), `updateQuery`, `resetQuery` 함수를 만들어서 URL 쿼리 파라미터를 업데이트/초기화 할 수 있도록 했습니다.

### 3. 컴포넌트 구조

초기에 `domain / shared`로 도메인 간 결합도, 의존도를 낮추고자 했는데, 진행하다보니 프로젝트 성격상 필요성보다 복잡도가 더 큰 구조라고 생각되어 루트에 각 디렉토리를 위치시키는 방식으로 변경했습니다.

```text
src/
├── apis/
├── components/
│   ├── common/
│   ├── layout/
│   ├── product/
│   └── ui/
├── constants/
├── hooks/
├── mocks/
├── styles/
├── types/
└── utils/
```

### 4. 검색 및 자동완성 구현

디바운싱은 hook 레이어에 걸었습니다. `useSearch`에서 입력값을 관리하고, `useDebounce`로 500ms 지연된 값을 만든 뒤 자동완성 API를 호출합니다.
UI 컴포넌트는 가능한 한 단순하게 유지합니다.

자동완성 드롭다운의 열기/닫기는 `Autocomplete` 내부에서 관리합니다. 입력 중이거나 포커스가 있을 때 목록을 열고, 외부 클릭, `Escape`, `Tab`, 항목 선택 시 닫히게 했습니다. 키보드 접근성은 `ArrowDown`, `ArrowUp`, `Enter`, `Escape`, `Tab` 기준으로 넣었습니다. 다만 현재는 "자동완성 API"에만 디바운스를 적용했고, 실제 상품 검색은 버튼 클릭 또는 제안 선택 시 반영되도록 구현되어 있습니다.

### 5. 에러/재시도 처리

비동기 요청을 담당하는 `AsyncBoundary`를 만들고, `ErrorBoundary`와 `Suspense`를 조합하여 에러와 로딩을 처리했습니다.
에러바운더리에서 `DefaultError` 컴포넌트를 보여주는 경우 `useQueryErrorResetBoundary`를 연결해 사용자가 "다시 시도하기" 버튼을 눌러 `reset` 동작을 할 수 있도록 했습니다.

### 6. 과제를 수행하면서 어려웠던 점

- `AutoComplete` 를 직접 구현하려고 하니 쉽지 않았습니다.
- 과제 진행 경험이 많이 없어 설계, 구조에 대한 고민에 시간이 많이 들었던 것 같습니다.

### 7. 추가 라이브러리

- `@tanstack/react-query`: 서버 상태를 관리하고 캐싱과 Suspense 활용을 위해 사용했습니다.
- `react-router-dom`: URL 쿼리스트링 기반 상태 관리를 위해 사용했습니다.
- `@emotion/react`: 직관적인 CSS 기반 스타일링을 위해 사용했습니다.
- `ky`: `fetch` 기반이지만 API 호출 코드를 좀 더 간결하게 만들 수 있어서 사용했습니다.
- `react-error-boundary`: 에러 UI와 재시도 흐름을 공통화하기 위해 사용했습니다.

### 8. 아쉬운 점 / 개선하고 싶은 점

- 생각보다 시간이 많이 걸려서 디테일하게 작업하지 못한 부분이 많아 아쉽습니다.

- 코드리뷰 편의성을 위해 되도록 한 곳에서 많은 컴포넌트를 보여주려 했는데, 오히려 가독성이 떨어질 수 있겠다는 생각이 들었습니다.

- `AutoComplete`, `Button` 과 같은 UI 코드는 빠른 구현을 위해 AI 우선 구현 후 리뷰 및 수정을 진행하려 했으나 시간 관계상 하지 못했습니다.

- `useSearch`, `useCategory` 등 컴포넌트에서 사용하는 로직을 훅으로 분리하는 과정에서 각 커스텀 훅이 순수하지 않다는 생각이 들었습니다. route, queryParam 관련 값들을 해당 커스텀 훅 내부에서 import 해오는 것이 아니라, UI 컴포넌트 레벨에서 매개변수 형태로 전달하는 것이 함수(훅)을 순수하게 유지하는 방법이 아닐까 하는 생각이 들었습니다.
